## 4. `organ.json` 内容（仅肾脏器官存在性检测）

json

```
[
    {
        "image_name": "002_1.jpg",
        "organ_present": true,
        "confidence": 0.91
    },
    {
        "image_name": "004_1.jpg",
        "organ_present": false,
        "confidence": 0.99
    }
]
```



说明：

- `002_1.jpg` 有器官命中 → 可进页面二。
- `004_1.jpg` 明确无肾脏，但它是已标注图片（在GT里），所以不会走页面二筛选，此处仅示例 organ 数据。
- `case_003` 对应的器官（肾脏）假设**没有 organ.json**，任务生成时跳过器官筛选直接进入页面二。

## 5. 这些数据覆盖的场景验证

| 场景                       | 图片        | 说明                                                         |
| :------------------------- | :---------- | :----------------------------------------------------------- |
| **困难样本**               | `001_1.jpg` | GT 是 `cyst`，预测是 `tumor` 且置信度 0.92，类别不同 → 进入页面一 |
| 非困难样本（对照）         | `001_2.jpg` | GT 与预测类别相同、IoU ≈ 0.97，一致 → 不进入页面一           |
| **无标注 + organ 命中**    | `002_1.jpg` | 不在 GT 中，`organ.json` 中 `organ_present=true`，有预测 → 进入页面二 |
| **无标注 + 无 organ.json** | `003_1.jpg` | 不在 GT 中，该器官无 `organ.json`，有预测 → 直接进入页面二   |
| **异常：预测缺失**         | `004_1.jpg` | GT 中有，但 `pred.json` 里没有该图预测 → 提示“模型预测信息缺失” |
| **异常：报告图缺失**       | `case_004`  | 文件夹无报告图 → 页面显示“暂无病例报告图片”                  |





####  其他信息



```
1. JSON 结构：
- GT.json 和 pred.json 单图允许多框，bbox 用 [x1, y1, x2, y2] 像素坐标。
- pred.json 与 GT.json 完全同构，额外包含 model_version 和 confidence。
- organ.json 字段：image_name, organ_present(bool), confidence(float)。

2. 困难样本筛选规则：
- 置信度阈值默认 0.85。
- 按类别不同 或 框最大 IoU < 0.5 为不一致。多框用贪婪匹配。

3. 登录：用本地 JSON 存账号，Flask session 认证，预设 admin/doctor。

4. 路径：任务清单存绝对路径，代码用 pathlib 统一为 /。case_id 取父目录名。

5. COCO 输出：类别映射手动定义，按标准 COCO 格式生成。
```

```
1. 类别映射：先写死一个字典，key=类别名, value=自增int。
2. COCO ID：image.id 和 annotation.id 都从1自增，不派生。
3. 账号：测试用明文存 JSON，生产用 werkzeug.security 哈希。
4. 任务分配：用任务池模式，JSONL 加 assigned_to 和 status 字段，医生顺序取下一条 pending。
5. 阈值：放 config.json 文件里可配置，不做管理界面。
6. 并发：用 threading.Lock 保护 JSONL 写入。
```
