# 医学影像标注检查系统需求文档 (MedRev)

## 1. 系统目标

系统用于医生快速审核两类模型相关样本：

1. **困难样本 GT 检查**  
   系统筛选模型高置信度但与原始 GT 不一致的已标注图片，医生判断该图片的 **GT 是否正确**。

2. **无标注图片伪标签检查**  
   系统筛选有伪标签价值的无标注图片，医生判断模型伪标签结果是否正确。

系统本地部署，所有数据均存储在本地服务器，不依赖云端存储。

---

## 2. 角色与登录

系统包含两类角色。

### 2.1 医生

医生登录后进入审核页面，只负责查看样本并提交审核结果。

医生权限：

- 查看分配给自己的审核任务；
- 查看病例报告图片、当前图片、GT 展示图、模型预测展示图、同病例其他图片；
- 页面一提交“GT 正确 / GT 有误”；
- 页面二提交“伪标签正确 / 伪标签错误”；
- 不需要进行数据导入、任务生成或结果汇总。

### 2.2 研究人员 / 管理员

研究人员或管理员负责数据准备、任务生成和结果导出。

管理员权限：

- 配置本地数据路径；
- 导入或读取器官级 JSON；
- 生成页面一、页面二任务清单；
- 查看审核进度；
- 导出审核结果；
- 导出 GT 问题清单；
- 生成可用于训练的伪标签清单；
- 管理不同模型版本和不同审核轮次。

### 2.3 登录要求

登录后系统至少需要记录：

```json
{
  "user_id": "string",
  "user_name": "string",
  "role": "doctor | admin"
}
```

医生提交审核结果时，需要将 `user_id` 保存为 `reviewer_id`。

## 3. 本地数据组织

### 3.1 原始病例目录

原始数据按器官和病例组织，一个病例一个文件夹。

```text
数据根目录/
  肾脏器官/
    年龄疾病类型等分级目录/
      001/
        病例报告.jpg
        001_1.jpg
        001_1.json
        001_2.jpg
        001_3.jpg
  其他器官/
    xxx/
```

说明：

- 每个病例文件夹下包含病例报告图片；
- 每个病例文件夹下包含多张超声图片；
- 有人工标注的图片存在对应 JSON；
- 无人工标注的图片没有对应 JSON；
- 同一病例下的其他图片用于医生多帧参考。

### 3.2 器官级 JSON 目录

为简化系统读取逻辑，提前将分散的 JSON 整理为按器官划分的统一文件。

```text
整理后标注目录/
  肾脏/
    GT.json
    organ.json
    pred.json
  肝脏/
    GT.json
    pred.json
```

| 文件 | 作用 |
|---|---|
| `GT.json` | 当前器官下所有已标注图片的人工 GT 信息 |
| `organ.json` | 当前器官的存在性检测结果，用于筛选哪些无标注图片值得进入页面二，可选 |
| `pred.json` | 模型对当前器官所有图片的标签预测信息 |

说明：

- `GT.json` 用于判断图片是否已有人工标注；
- `organ.json` 只用于判断图片中是否存在当前器官，不是伪标签结果；
- 如果某个器官没有 `organ.json`，则跳过器官存在性筛选；
- `pred.json` 用于页面一模型预测展示，以及页面二伪标签展示。

## 4. 样本导入与任务生成

系统不在页面中动态判断哪些样本需要审核，而是通过离线脚本提前生成任务清单。

### 4.1 页面一任务生成：困难样本 GT 检查

输入文件：

- `GT.json`
- `pred.json`

生成逻辑：

1. 从 `GT.json` 读取当前器官所有已标注图片；
2. 从 `pred.json` 读取这些图片对应的模型预测结果；
3. 筛选模型高置信度但与 GT 不一致的图片；
4. 生成页面一待审核任务。

输出文件：

```text
tasks/hard_sample_gt_review.jsonl
```

### 4.2 页面二任务生成：无标注图片伪标签检查

输入文件：

- `GT.json`
- `organ.json`，可选
- `pred.json`

生成逻辑：

1. 先根据 `GT.json` 判断当前图片是否为无标注图片；
2. 如果图片已经存在于 `GT.json`，说明该图片已有人工标注，不进入页面二；
3. 如果图片不在 `GT.json` 中，说明该图片为无标注图片；
4. 如果该器官存在 `organ.json`，则继续判断该图片是否检测到当前器官；
5. 如果 `organ.json` 显示该图片没有检测到当前器官，则跳过；
6. 如果 `organ.json` 显示该图片检测到当前器官，则进入页面二；
7. 如果该器官没有 `organ.json`，则跳过器官存在性筛选，直接根据 `pred.json` 中的伪标签结果生成页面二任务；
8. 页面二展示 `pred.json` 中的伪标签结果，由医生判断伪标签是否正确。

输出文件：

```text
tasks/pseudo_label_review.jsonl
```

## 5. 页面一：困难样本 GT 检查

### 5.1 页面用途

用于医生判断困难样本的原始 GT 是否正确。

样本来源：

- 图片已有人工 GT；
- 模型预测置信度较高；
- 模型预测结果与 GT 不一致。

### 5.2 页面展示内容

页面展示：

1. **病例报告图片**
   - 读取本地病例报告图片；
   - 支持放大、缩小、拖拽查看；
   - 图片缺失时显示“暂无病例报告图片”。

2. **当前审核图像**
   - 展示当前困难样本原图；
   - 支持放大、缩小、拖拽查看。

3. **GT 展示图**
   - 展示当前图片对应的 GT 标注结果；
   - 可由后端提前渲染为带 GT 的图像。

4. **模型预测展示图**
   - 展示当前图片对应的模型预测结果；
   - 可由后端提前渲染为带预测框 / 标签的图像；
   - 展示预测类别、置信度、模型版本。

5. **同病例其他图片**
   - 展示该病例下的其他超声图片；
   - 支持横向滑动查看；
   - 点击图片后可放大查看。

### 5.3 医生操作

医生只需要选择以下结果之一：

1. **GT 正确**
2. **GT 有误**

不要求医生手动输入文字备注。

---

## 6. 页面二：无标注图片伪标签检查

### 6.1 页面用途

用于医生判断无标注图片的模型伪标签是否正确。

样本来源：

- 图片不在 `GT.json` 中，即无人工标注；
- 如果存在 `organ.json`，该图片需要通过当前器官的存在性筛选；
- 如果不存在 `organ.json`，则直接使用 `pred.json` 生成任务；
- 该图片存在模型伪标签预测结果。

### 6.2 页面展示内容

页面展示：

1. **病例报告图片**
   - 读取本地病例报告图片；
   - 支持放大、缩小、拖拽查看；
   - 图片缺失时显示“暂无病例报告图片”。

2. **当前审核图像**
   - 展示当前无标注图片原图；
   - 支持放大、缩小、拖拽查看。

3. **模型伪标签展示图**
   - 展示模型对该图片的伪标签结果；
   - 伪标签信息来自 `pred.json`；
   - 展示类别、置信度、模型版本。

4. **同病例其他图片**
   - 展示该病例下的其他超声图片；
   - 支持横向滑动查看；
   - 点击图片后可放大查看。

说明：

- `organ.json` 只用于离线筛选是否进入页面二；
- 页面二不必须展示 `organ.json` 的器官检测结果；
- 页面二主要展示 `pred.json` 中的伪标签预测结果。

### 6.3 医生操作

医生先判断伪标签是否正确：

1. **伪标签正确**
2. **伪标签错误**

如果选择“伪标签错误”，需要从以下固定错误类型中选择：

| 错误类型 | 含义 |
|---|---|
| 类别错误 | 预测类别不正确 |
| 框大小错误 | 框中心位置基本正确，但框过大或过小 |
| 误检 | 模型检测出了不应存在的目标 |
| 漏检 | 模型遗漏了应检测出的目标 |

不要求医生手动输入文字备注。

## 7. 本地文件访问要求

所有图片和 JSON 均存储在本地服务器。

要求：

- 任务清单中保存本地文件路径，不保存云端 URL；
- 前端不直接读取本地磁盘路径；
- 后端根据任务中的本地路径读取图片和 JSON，并返回给页面展示；
- 审核结果保存到本地服务器；
- 医生只能访问被分配任务范围内的本地文件。

## 9. 审核结果沉淀建议

审核结果建议分为两层：

1. **原始审核记录**：保存医生每次审核点击结果，用于追溯；
2. **派生结果清单**：从原始审核记录自动生成，用于 GT 返修、伪标签训练和模型迭代。

### 9.1 推荐输出目录

```text
review_outputs/
  run_id/
    raw_reviews.jsonl
    gt_issue_list.jsonl
    accepted_pseudo_labels.jsonl
    accepted_pseudo_labels.coco.json
    pseudo_label_error_list.jsonl
    summary.json
```

### 9.2 原始审核记录：raw_reviews.jsonl

原始记录字段需要足够追溯，但不必重复保存病例报告路径、完整 GT、完整预测等大对象。

```json
{
  "run_id": "string",
  "task_id": "string",
  "task_type": "hard_sample_gt_review | pseudo_label_review",
  "organ": "kidney",
  "image_name": "001_1.jpg",
  "image_path": "本地原图路径",
  "review_result": "gt_correct | gt_error | pseudo_label_correct | pseudo_label_error",
  "error_type": "category_error | box_size_error | false_positive | false_negative | null",
  "reviewer_id": "string",
  "submitted_at": "ISO-8601 datetime"
}
```

说明：

- 页面一只会产生 `gt_correct` 或 `gt_error`；
- 页面二只会产生 `pseudo_label_correct` 或 `pseudo_label_error`；
- `error_type` 只在页面二选择 `pseudo_label_error` 时填写；
- 完整标注内容可以通过 `run_id + task_id` 回查任务清单，不需要在原始审核记录中重复保存。

### 9.3 GT 问题清单：gt_issue_list.jsonl

医生在页面一选择 **GT 有误** 的样本进入该文件。

```json
{
  "run_id": "string",
  "task_id": "string",
  "organ": "kidney",
  "image_name": "001_1.jpg",
  "image_path": "本地原图路径"
}
```

用途：

- 记录医生认为 GT 有误的样本；
- 方便研究人员后续查看和统计 GT 问题；
- 当前版本不支持在线返修，也不要求系统自动修改原始 GT 或重新生成 `GT.json`。

说明：

- GT 返修、标注修改、新版 `GT.json` 生成可作为后续功能扩展。

### 9.4 可用伪标签清单：accepted_pseudo_labels.jsonl

医生在页面二选择 **伪标签正确** 的样本进入该文件。

```json
{
  "run_id": "string",
  "task_id": "string",
  "organ": "kidney",
  "image_name": "001_2.jpg",
  "image_path": "本地原图路径",
  "pseudo_label_prediction": "object"
}
```

用途：

- 作为新增训练样本；
- 可转换为 `accepted_pseudo_labels.coco.json`；
- 后续可与修正后的 GT 一起组成新训练数据版本。

### 9.5 伪标签错误清单：pseudo_label_error_list.jsonl

医生在页面二选择 **伪标签错误** 的样本进入该文件。

```json
{
  "run_id": "string",
  "task_id": "string",
  "organ": "kidney",
  "image_name": "001_2.jpg",
  "image_path": "本地原图路径",
  "error_type": "category_error | box_size_error | false_positive | false_negative"
}
```

用途：

- 统计模型错误类型；
- 分析模型在不同器官上的问题；
- 为下一轮模型优化提供依据。

### 9.6 汇总文件：summary.json

```json
{
  "run_id": "string",
  "total_reviewed": 0,
  "gt_error_count": 0,
  "accepted_pseudo_label_count": 0,
  "pseudo_label_error_count": 0,
  "error_type_count": {
    "category_error": 0,
    "box_size_error": 0,
    "false_positive": 0,
    "false_negative": 0
  }
}
```

## 10. 闭环 Pipeline

每一轮模型推理和医生审核都使用独立 `run_id`。

`run_id` 建议包含：

- 器官名称；
- 模型版本；
- 推理日期；
- 数据版本。

示例：

```text
kidney_model_v3_20250101_data_v2
```

闭环流程：

```text
原始数据 + GT.json
        ↓
模型推理生成 pred.json
        ↓
可选 organ.json 做器官存在性筛选
        ↓
生成页面一 / 页面二任务清单
        ↓
医生审核
        ↓
生成 GT 问题清单 + 可用伪标签清单
        ↓
后续可选 GT 返修 + 合并医生确认的伪标签
        ↓
形成新训练数据版本
        ↓
训练新模型
        ↓
重新推理，进入下一轮闭环
```

---

## 11. 异常处理要求

| 异常情况 | 页面处理 |
|---|---|
| 病例报告图片缺失 | 显示“暂无病例报告图片” |
| 当前图片加载失败 | 显示加载失败，并支持重试 |
| 同病例其他图片为空 | 显示“暂无其他图片” |
| GT 信息缺失 | 页面一提示“GT 信息缺失” |
| 模型预测信息缺失 | 显示“模型预测信息缺失” |
| JSON 保存失败 | 提示保存失败，允许重试，不进入下一条 |

---

## 12. 验收标准

1. 系统支持医生和管理员两类登录角色；
2. 管理员可基于本地病例目录和器官级 JSON 生成任务；
3. 页面一可基于 `GT.json + pred.json` 生成 GT 检查任务；
4. 页面二可先通过 `GT.json` 判断无标注图片，再结合可选 `organ.json` 和 `pred.json` 生成伪标签检查任务；
5. 如果某个器官没有 `organ.json`，系统可跳过器官存在性筛选；
6. 页面一可展示病例报告图片、当前图像、GT 展示图、模型预测展示图和同病例其他图片；
7. 页面二可展示病例报告图片、当前图像、模型伪标签展示图和同病例其他图片；
8. 医生在页面一只需选择“GT 正确”或“GT 有误”；
9. 医生在页面二选择“伪标签错误”时，只需从固定错误类型中选择；
10. 审核结果保存为本地 JSON / JSONL；
11. 系统可自动生成 GT 问题清单；
12. 系统可自动生成医生确认可用的伪标签训练清单；
13. 每轮结果通过 `run_id`、`task_id`、`organ`、`image_name` 追踪，支持后续模型迭代闭环；
14. 保存成功后自动进入下一条样本；
15. 保存失败时保留当前页面并提示重试；
16. 系统通过同一个本地 Web 链接支持 PC、平板、手机访问。
