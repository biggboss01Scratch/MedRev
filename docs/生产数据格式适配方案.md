# 实际生产数据分析与适配方案

## 概述

实际生产数据（Test 文件夹）采用**源数据形式**，而系统设计基于**工程化数据形式**。本文详细分析两者的差异，并提出数据转换与适配方案。

---

## 第 1 部分：数据结构对比

### 1.1 原需求文档中的数据结构

#### 原始病例目录

```
数据根目录/
  肾脏/
    年龄分组/
      001/
        病例报告.jpg（单一文件）
        001_1.jpg
        001_1.json
        001_2.jpg
        001_3.jpg
  其他器官/
```

**特点**：
- 病例按**单一器官**组织（一个器官一个顶级目录）
- 每个病例一个"病例报告.jpg"
- 图像与标注一对一关联（有 .jpg 可能对应 .json）

#### 整理后标注目录（工程化）

```
整理后标注目录/
  肾脏/
    GT.json          # COCO 格式，所有已标注图片的 GT
    organ.json       # COCO 格式，器官存在性检测结果
    pred.json        # COCO 格式，模型伪标签预测
  肝脏/
    GT.json
    pred.json
```

**特点**：
- 集中式设计：所有信息聚合到 3 个文件
- COCO 格式标准化
- 易于系统读取和任务生成

## 第 2 部分：关键差异分析

### 2.1 组织结构差异

| 维度 | 需求文档 | 实际数据 | 影响 |
|-----|--------|--------|------|
| **器官组织** | 器官为顶级目录 | 诊断为顶级，器官隐含 | 需要通过目录名推断器官 |
| **病例分组** | 单层（病例ID） | 多层（诊断→年龄→性别→病例ID） | 需要处理多级目录查询 |
| **报告图片** | 单一 `病例报告.jpg` | 多个关联器官图 | 关联图片概念不同 |
| **标注形式** | 统一 JSON | 分散 ISAT 格式 JSON | 需要格式转换 |

### 2.2 数据形式差异

| 项目 | 需求文档 | 实际数据 |
|-----|--------|--------|
| **标注格式** | COCO（集中式） | ISAT（分散式） |
| **图像标注** | 按 image_id 在 GT.json 中汇总 | 每个图像一个 JSON 文件 |
| **标注覆盖** | 可能 100% | 仅 5.53% |
| **器官存在性** | organ.json（预测结果） | isat.yaml（类别定义） |
| **病例信息** | 隐含在路径 | 显式在 病史.txt、isat.yaml |

### 2.3 功能流程差异

**需求文档流程**：
```
整理后标注目录（GT.json/organ.json/pred.json）
  ↓
generate_tasks.py（读取 COCO 格式）
  ↓
tasks/hard_sample_gt_review.jsonl
tasks/pseudo_label_review.jsonl
```

**实际数据流程**（需新增）：
```
Test/raw_subset（ISAT 格式源数据）
  ↓
转换脚本（isat_to_coco.py）← 【需要编写】
  ↓
整理后标注目录（GT.json/organ.json/pred.json）
  ↓
generate_tasks.py（保持不变）
  ↓
tasks/hard_sample_gt_review.jsonl
tasks/pseudo_label_review.jsonl
```

### 2.4 标识更明确的原因

在新的实际数据下，`run_id` 和 `task_id` 反而比旧测试数据更容易定义清楚，因为源数据天然带有更完整的路径语义和病例上下文。

| 标识 | 作用 | 新数据下的推荐定义 |
|-----|------|------------------|
| `run_id` | 一轮任务生成和审核的批次号 | 表示一次导入、一次转换或一次模型推理批次，不要直接等同于单个病例 |
| `task_id` | 单条审核任务的唯一标识 | 直接由任务类型 + 诊断 + 病例路径 + 图像相对路径构成，保证唯一且可追溯 |

#### 推荐含义

- `run_id` 应该代表“这一次生成任务的整体版本”，例如数据版本、转换版本、模型版本、生成日期的组合。
- `task_id` 应该代表“这一张图在这一轮里的一条具体审核任务”，并且最好能从原始目录反推回去。

#### 推荐格式

```text
run_id = data_v1_convert_v1_model_v3_20260507
task_id = hard|pseudo + 诊断 + 病例相对路径 + 图像文件名
```

例如：

```text
run_id = raw_subset_v1_convert_v1_20260507
task_id = hard_肾结石≥0.5结石(尿液分析)_50岁以下(315例)_女性159_肾脏_p0165_肾脏_p0165_022_144122.jpg
```

#### 为什么更明确

- 新数据的目录本身包含诊断、年龄、性别和病例号，`task_id` 不需要再依赖模糊的人工编号。
- 图像文件名本身已经包含病例前缀和序列号，适合直接作为任务子标识。
- 后续做任务回查时，只要知道 `run_id + task_id`，就能定位到具体源文件和对应 JSON。
- 这比旧测试数据里的“简单病例目录 + 统一整理文件”更适合做审计和回溯。

#### 实践建议

- `run_id` 由系统生成或管理员显式输入，不建议从单个病例推断。
- `task_id` 由脚本稳定生成，不建议人工编辑。
- 如果后续要支持多轮复核，`task_id` 中还可以加入版本前缀，但要保持同一轮内稳定不变。

## 第 3 部分：适配方案

### 3.1 方案概述

采用**分层适配**策略：

| 层级 | 目标 | 方式 |
|-----|------|------|
| **第 1 层：数据采集** | 扫描源数据 | 编写脚本扫描 Test 目录，收集文件信息 |
| **第 2 层：格式转换** | ISAT → COCO | 编写 `isat_to_coco.py` 脚本转换标注 |
| **第 3 层：数据聚合** | 分散 JSON → 集中文件 | 聚合所有病例的标注为 GT.json/organ.json |
| **第 4 层：任务生成** | 保持现有逻辑 | `generate_tasks.py` 不变 |

### 3.2 新增脚本设计

#### 脚本 1：`scripts/isat_to_coco.py`

**功能**：
1. 扫描 `Test/raw_subset` 目录
2. 对每个病例：
   - 读取 `isat.yaml` 获取类别映射
   - 读取各 JSON 文件中的标注
   - 转换为 COCO 格式
3. 输出 `整理后标注目录/肾脏/GT.json`（所有有标注的图像）

**输入**：
```
Test/raw_subset/[诊断]/[年龄]/[性别]/[病例ID]/
  ├─ isat.yaml
  └─ *.json（标注文件）
```

**输出**：
```
整理后标注目录/肾脏/
  └─ GT.json（COCO 格式）

{
  "images": [
    {
      "id": 1,
      "file_name": "肾脏_p0165_022_144122.jpg",
      "width": 1290,
      "height": 900,
      "image_relpath": "肾结石≥0.5结石(尿液分析)/50岁以下(315例)/女性159/肾脏_p0165/肾脏_p0165_022_144122.jpg",
      "image_abspath": "D:/Projects/MedRev/Test/raw_subset/...",
      "case_relpath": "肾结石≥0.5结石(尿液分析)/50岁以下(315例)/女性159/肾脏_p0165",
      "organ": "肾脏",
      "diagnosis": "肾结石≥0.5结石(尿液分析)",
      "age_group": "50岁以下(315例)",
      "gender": "女性"
    }
  ],
  "annotations": [
    {
      "id": 1,
      "image_id": 1,
      "category_id": 1,
      "bbox": [611.32, 394.99, 91.25, 68.68],  // 转换为 [x, y, w, h]
      "area": 6108.38,
      "iscrowd": 0,
      "segmentation": [[...]]
    }
  ],
  "categories": [
    {"id": 1, "name": "肾结石", "supercategory": "肾脏"},
    {"id": 2, "name": "肾结石积水", "supercategory": "肾脏"}
  ]
}
```

#### 脚本 2：`scripts/collect_related_images.py`

**功能**：
1. 扫描每个病例目录
2. 收集关联图片（下肢动脉、下肢静脉、泌尿系等）
3. 生成 `病例元数据.json` 供任务生成时查询

**输出**：
```
整理后标注目录/肾脏/
  └─ case_metadata.json

{
  "肾脏_p0165": {
    "report_images": [
      "下肢动脉.jpg",
      "下肢静脉.jpg",
      "泌尿系.jpg"
    ],
    "case_abspath": "D:/Projects/MedRev/Test/raw_subset/.../肾脏_p0165",
    "diagnosis": "肾结石≥0.5结石(尿液分析)",
    "age_group": "50岁以下(315例)",
    "gender": "女性"
  }
}
```

#### 脚本 3：修改 `scripts/generate_tasks.py`

**改动点**：
1. 支持从 `case_metadata.json` 读取关联图片
2. 改进 `find_image_path` 逻辑，使用 `image_relpath` 作为主键
3. 支持处理多个报告图片

## 第 6 部分：总结

### 核心要点

1. **数据形式差异**：实际数据采用分散的源数据形式（ISAT），需求文档要求集中的工程化形式（COCO）。

2. **适配策略**：通过新增两个转换脚本 + 对 generate_tasks.py 的小改动，将源数据无缝转换为系统可用的格式。

3. **系统逻辑不变**：核心的任务生成、审核、导出逻辑保持不变，确保向后兼容。

4. **可接受的差异**：
   - 多个报告图片 vs 单一报告图片 → 前端可支持列表展示
   - 低标注覆盖率（5.53%）→ 符合无标注图片检查的需求场景

5. **推荐执行顺序**：
   1. 编写 isat_to_coco.py（优先级最高）
   2. 编写 collect_related_images.py
   3. 修改 generate_tasks.py
   4. 集成测试
   5. 可选：前端优化以展示更多诊断信息
