# MedRev 项目 Agent 提示文档

注：该文档为我和deepseek讨论生成，仅供参考。“标注系统需求文档”更为核心。

## 项目名称

**MedRev**（Medical Annotation Review System）
简述：医学影像标注审核系统，医生审核模型困难样本的 GT 是否正确，以及无标注图片的伪标签是否正确。

## 项目背景与核心概念

| 术语               | 含义                                                         |
| :----------------- | :----------------------------------------------------------- |
| GT（Ground Truth） | 医生人工标注的"标准答案"，包含病灶位置坐标和类别             |
| 伪标签             | 模型对**无人工标注**图片自行推断的标注结果，不一定正确，需医生确认 |
| 困难样本           | 模型高置信度预测与人工 GT 不一致的图片，需医生判断是 GT 错了还是模型错了 |
| 器官存在性检测     | 通过 `organ.json` 判断图片是否拍到目标器官，用于初筛无标注图片 |
| 病例               | 一个患者一次超声检查产生的全部资料，一个文件夹包含1张报告图+N张超声图 |

## 系统架构要点

- **本地部署**：所有数据和图片存储在本地服务器，无云端依赖
- **两个角色**：
  - **医生**：仅审核分配到的任务，点击选项提交结果，无文本输入
  - **管理员**：配置路径、触发生成任务、查看进度、导出结果
- **两个审核页面**：
  - 页面一「困难样本 GT 检查」：判断 GT 是否正确（GT正确 / GT有误）
  - 页面二「无标注图片伪标签检查」：判断伪标签是否正确（正确 / 错误，错误需选类型：类别错误/框大小错误/误检/漏检）
- **任务离线生成**：管理员通过脚本提前生成任务清单（`.jsonl`），医生审核时不实时计算
- **闭环迭代**：每轮审核用唯一 `run_id` 标识，审核结果可反馈至下一轮模型训练

## 关键数据文件结构

### 原始病例目录（只读图片，不读其JSON）

text

```
原始数据根目录/
  {器官名}/
    {分级目录}/
      {病例文件夹}/
        病例报告.jpg
        xxx_1.jpg
        xxx_1.json     ← 系统不读这些分散的JSON
        xxx_2.jpg
        ...
```



### 整理后标注目录（管理员提前汇总，系统只读）

text

```
整理后标注目录/
  {器官名}/
    GT.json       ← 该器官所有已标注图片的人工标注
    organ.json    ← 器官存在性检测结果（可选）
    pred.json     ← 模型对该器官所有图片的预测
```



- `GT.json`：用于判断图片是否已有人工标注
- `organ.json`：仅用于筛选无标注图片中是否拍到目标器官
- `pred.json`：用于页面一模型预测展示、页面二伪标签展示

### 系统工作目录（系统自动生成）

text

```
tasks/
  hard_sample_gt_review.jsonl      ← 页面一任务清单
  pseudo_label_review.jsonl        ← 页面二任务清单
review_outputs/
  {run_id}/
    raw_reviews.jsonl              ← 原始审核记录
    gt_issue_list.jsonl            ← GT有误样本清单
    accepted_pseudo_labels.jsonl   ← 医生确认可用的伪标签
    accepted_pseudo_labels.coco.json
    pseudo_label_error_list.jsonl  ← 伪标签错误样本及错误类型
    summary.json                   ← 统计汇总
```



## 技术栈选型结论

采用**轻量全栈方案**：

| 层       | 技术                                      |
| :------- | :---------------------------------------- |
| 后端     | Python + Flask                            |
| 前端     | Jinja2 模板 + Bootstrap 5（响应式）       |
| 图片渲染 | Pillow（后端在图片上画标注框后返回 JPEG） |
| 数据存储 | JSON/JSONL 文件，无传统数据库             |
| 用户登录 | Flask-Login + Session                     |

关键原则：

- **图片渲染必须后端做**（前端只用 `<img>` 标签展示，不画框）
- 任务生成脚本与审核页面解耦
- 审核结果写入 JSONL，通过 `run_id` + `task_id` 回溯完整信息

## 你需要做什么（排除项）

- ❌ 不需要调用任何 AI 模型（pred.json、organ.json 都是提前生成好的）
- ❌ 不需要遍历原始病例目录并合并分散的 JSON（管理员已提前汇总）
- ❌ 不需要让医生输入任何自由文本（仅点击选项）
- ✅ 需要从原始病例目录读取原图、病例报告图、同病例其他图
- ✅ 需要从整理后标注目录读取 GT、预测、器官检测结果
- ✅ 需要展示图片（原图 + 后端渲染的标注图）
- ✅ 需要实现任务生成脚本（读取现有 JSON 生成任务清单）
- ✅ 需要实现审核提交逻辑（写入 JSONL，自动跳下一条）

## 开发建议

1. 先定义好 `GT.json`、`pred.json`、`organ.json` 的内部字段结构
2. 先写任务生成脚本，这是核心逻辑（能吐出正确的任务清单就完成了大半）
3. 后端图片渲染用 Pillow，接收坐标数组，画矩形和文字标签
4. 写测试文件（重点测任务生成逻辑和审核结果写入逻辑）
5. 异常处理：图片缺失显示占位文字，JSON 保存失败提示重试不跳转
